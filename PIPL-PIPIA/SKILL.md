### 步骤1.5——**多份报告判定逻辑（关键）**

#### 核心概念：触发情形 vs 处理活动分组

```
┌─────────────────────────────────────────────────────────────────────┐
│                          PIPIA报告结构                               │
├─────────────────────────────────────────────────────────────────────┤
│  第一层级：触发情形（PIPL §55）                                       │
│  ├── 触发情形（一）处理敏感个人信息 → 独立报告1                       │
│  ├── 触发情形（二）自动化决策 → 独立报告2                            │
│  └── 触发情形（三）委托处理/向他人提供/公开 → 独立报告3               │
│                                                                      │
│  第二层级：处理活动分组（在每份报告内部）                              │
│  └── 按事实性逻辑分组：处理目的、信息类型、处理方式等                  │
└─────────────────────────────────────────────────────────────────────┘
```

**错误理解**：每个触发情形 = 一组处理活动 ❌
**正确理解**：每个触发情形 = 一份独立报告；报告内部再按事实性逻辑分组处理活动 ✓

#### 多情形判定流程

```
触发情形计数 → 判定结果
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
仅触发1个情形  → 生成1份PIPIA报告
触发2个情形    → 生成2份独立PIPIA报告
触发N个情形    → 生成N份独立PIPIA报告
```

#### ⛔ 常见违规思维模式（必须避免）

| ❌ 违规思维 | 正确理解 |
|------------|----------|
| "触发情形较多，先输出一份综合评估报告，后续如需可分别为各触发情形生成独立报告" | **禁止**：绝不允许以"后续可分"为由先生成综合报告。必须立即生成独立报告。 |
| "先综合分析所有风险点，再拆分报告" | **禁止**：每个触发情形的评估是独立的法定义务。 |
| "一份报告更便于用户阅读" | **禁止**：法定要求高于便利性。 |
| "时间有限，先快速生成一份总览" | **禁止**：无论任何原因，都不允许生成综合报告替代独立报告。 |

**核心原则**：
- **不存在"过渡方案"**：没有"先综合、后分份"的中间状态
- **不存在"后续可分"**：一旦生成综合报告，就是违规
- **不存在"用户选择"**：这是法定要求，不是用户可选的交付方式

#### 多情形判定输出格式

完成步骤1分析后，**必须立即输出以下判定表**：

```markdown
## 多情形判定结果

| # | 触发情形 | 是否触发 | 触发依据原文 | 报告要求 |
|---|----------|----------|--------------|----------|
| 1 | 处理敏感个人信息 | 是/否 | [引用§55原文] | 如触发，需生成报告 |
| 2 | 自动化决策 | 是/否 | [引用§55原文] | 如触发，需生成报告 |
| 3 | 委托处理/向他人提供/公开 | 是/否 | [引用§55原文] | 如触发，需生成报告 |
| 4 | 向境外提供 | 是/否 | [引用§55原文] | 如触发，需生成报告 |
| 5 | 对个人权益有重大影响 | 是/否 | [引用§55原文] | 如触发，需生成报告 |

**触发情形总数**：[N]个
**需生成报告总数**：[N]份
```

### 步骤1——收集信息

### 步骤1.5——**信息完整性检查（关键）**

### 步骤3——数据映射（§5.3）

按 GB/T 39335 附录C格式创建数据映射，**针对每组处理活动分别创建数据映射表**。

### 步骤4——风险矩阵分析（核心）

**4×4风险矩阵分析**：针对每组处理活动，进行**4（风险源）× 4（权益影响）= 16项组合判定**。

**详细方法论**：见 `references/risk-matrix-methodology.md`

**核心要点**：
- 安全事件可能性判定 → 强制引用附录D.1
- 影响程度判定 → 强制引用附录D.2
- 风险等级判定 → 强制引用附录D.3

### 步骤5——各处理活动风险汇总

对每组处理活动统计风险等级分布。

### 步骤6——起草报告

使用 `assets/pipia-template-blank.md` 作为可填写模板。

**报告结构**：法律依据与定义、处理活动分组与数据映射、风险矩阵分析（16项）、风险汇总、综合评估结论、差距与整改建议、整改验证计划。

**如涉及多情形**：为每个触发情形起草独立报告，明确标注该报告对应的触发情形。

### 步骤7——强制独立验证

**关键**：交付前执行工作流3进行独立验证。

### 步骤8——最终交付

验证后，交付置信声明或剩余差距披露。

## 工作流3：独立PIPIA验证

**所有PIPIA报告交付前强制执行。**

### 步骤

1. **保存草稿**：保存至 `verification-temp/draft-pipia-v<迭代号>.md`
2. **派生验证代理**：使用Agent工具，不携带起草会话上下文
3. **验证提示模板**：

```
你是PIPIA验证代理。独立验证一份PIPIA草稿报告。

**预验证步骤（强制）**：
1. 阅读 `references/verification-checklist.md`
2. Grep PIPL §55-56 并阅读完整原文
3. 阅读 GB/T 39335-2020 第5章
4. 按 `references/legal-source-roadmap.md` 阅读其他适用法律
5. 如有读取失败，停止并报告失败

**任务**：
1. 执行预验证步骤
2. 阅读草稿报告
3. 对照源文本比较每个检查清单项
4. 生成差距报告（已阅读源文件、汇总统计、详细差距表、验证结论、建议）
```

4. **接收差距报告**：处理验证代理的差距报告
5. **如发现差距则修订**：最多迭代3次
6. **交付**：如无差距则置信交付；如有剩余差距则按 `references/output-format-guidance.md` 披露
