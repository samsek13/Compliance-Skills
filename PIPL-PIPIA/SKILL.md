### 步骤1.5——**多份报告判定逻辑（关键）**

#### 核心概念：触发情形 vs 处理活动分组

```
┌─────────────────────────────────────────────────────────────────────┐
│                          PIPIA报告结构                               │
├─────────────────────────────────────────────────────────────────────┤
│  第一层级：触发情形（PIPL §55）                                       │
│  ├── 触发情形（一）处理敏感个人信息 → 独立报告1                       │
│  ├── 触发情形（二）自动化决策 → 独立报告2                            │
│  └── 触发情形（三）委托处理/向他人提供/公开 → 独立报告3               │
│                                                                      │
│  第二层级：处理活动分组（在每份报告内部）                              │
│  └── 按事实性逻辑分组：处理目的、信息类型、处理方式等                  │
└─────────────────────────────────────────────────────────────────────┘
```

**错误理解**：每个触发情形 = 一组处理活动 ❌
**正确理解**：每个触发情形 = 一份独立报告；报告内部再按事实性逻辑分组处理活动 ✓

#### 多情形判定流程

```
触发情形计数 → 判定结果
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
仅触发1个情形  → 生成1份PIPIA报告
触发2个情形    → 生成2份独立PIPIA报告
触发N个情形    → 生成N份独立PIPIA报告
```

#### ⛔ 常见违规思维模式（必须避免）

| ❌ 违规思维 | 正确理解 |
|------------|----------|
| "触发情形较多，先输出一份综合评估报告，后续如需可分别为各触发情形生成独立报告" | **禁止**：绝不允许以"后续可分"为由先生成综合报告。必须立即生成独立报告。 |
| "先综合分析所有风险点，再拆分报告" | **禁止**：每个触发情形的评估是独立的法定义务。 |
| "一份报告更便于用户阅读" | **禁止**：法定要求高于便利性。 |
| "时间有限，先快速生成一份总览" | **禁止**：无论任何原因，都不允许生成综合报告替代独立报告。 |

**核心原则**：
- **不存在"过渡方案"**：没有"先综合、后分份"的中间状态
- **不存在"后续可分"**：一旦生成综合报告，就是违规
- **不存在"用户选择"**：这是法定要求，不是用户可选的交付方式

#### 多情形判定输出格式

完成步骤1分析后，**必须立即输出以下判定表**：

```markdown
## 多情形判定结果

| # | 触发情形 | 是否触发 | 触发依据原文 | 报告要求 |
|---|----------|----------|--------------|----------|
| 1 | 处理敏感个人信息 | 是/否 | [引用§55原文] | 如触发，需生成报告 |
| 2 | 自动化决策 | 是/否 | [引用§55原文] | 如触发，需生成报告 |
| 3 | 委托处理/向他人提供/公开 | 是/否 | [引用§55原文] | 如触发，需生成报告 |
| 4 | 向境外提供 | 是/否 | [引用§55原文] | 如触发，需生成报告 |
| 5 | 对个人权益有重大影响 | 是/否 | [引用§55原文] | 如触发，需生成报告 |

**触发情形总数**：[N]个
**需生成报告总数**：[N]份
```

## 工作流2：PIPIA报告起草

当被要求起草PIPIA报告：

### 核心架构：层级化多代理并行执行

**本工作流采用层级化多代理架构，避免单代理承担过多工作量导致的质量下降问题。**

```
┌─────────────────────────────────────────────────────────────────────┐
│  层级化多代理架构                                                      │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  报告头代理（report-head）                                             │
│    ├─ 撰写章节1-3（基本信息、法律依据、处理活动分组与数据映射）            │
│    ├─ 确定处理活动分组数量（N组）                                        │
│    ├─ 并行派生N个风险评估子代理（report-sub）                            │
│    ├─ 接收子代理返回的风险矩阵章节                                       │
│    ├─ 集成章节并撰写章节7-10（综合结论、整改、验证、声明）                 │
│    └─ 派生验证代理                                                     │
│                                                                      │
│  风险评估子代理（report-sub，N个并行执行）                               │
│    ├─ 接收：处理活动描述、信息类型、相关上下文                            │
│    ├─ 独立阅读GB/T 39335-2020附录D原文                                 │
│    ├─ 完成完整16项风险矩阵分析                                          │
│    └─ 返回完成的章节给报告头代理                                         │
│                                                                      │
│  验证代理（review agent）                                              │
│    ├─ 对完整集成报告执行验证                                            │
│    └─ 生成差距报告                                                     │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

**关键原则**：
- **每组处理活动由独立子代理负责**：避免单代理处理多组时的疲劳问题
- **并行执行**：多个子代理同时工作，提升效率
- **上下文隔离**：子代理仅接收必要的处理活动描述，独立阅读源文件
- **单一验证**：对集成后的完整报告执行验证，保持现有验证流程

1. 使用 `references/legal-source-roadmap.md` 确定适用法律
2. 阅读 GB/T 39335-2020 §4-5 和附录C-D
3. 阅读 PIPL §55-56 以及行业/特殊主体法律（如适用）
4. 阅读 `references/pipia-report-template.md` 了解结构指引

#### 步骤6.2——验证迭代协议

```
验证迭代流程：
├── 第1次验证 → 发现差距 → 修订报告 → 第2次验证
├── 第2次验证 → 发现差距 → 修订报告 → 第3次验证
├── 第3次验证 → 发现差距 → 记录剩余差距 → 交付
└── 验证通过 → 无差距 → 置信交付

⛔ 差距分类与处理强制要求：
| 差距类型 | 定义 | 处理要求 |
|---------|------|---------|
| 信息缺失型 | 需运营主体提供的事实信息（如具体存储期限、加密技术细节） | 可披露为剩余差距，标注"需运营主体核实" |
| 整改型 | 可通过补充分析、规范引用格式、完善章节结构等方式修复 | **禁止披露为剩余差距，必须整改后重新验证** |

禁止行为：
❌ 以"时间成本"、"边际价值"、"非关键项"为由跳过整改型差距
❌ 将整改型差距归类为信息缺失型
❌ 在整改型差距未清零时交付报告

⛔ 禁止行为：
❌ 跳过验证直接交付
❌ 仅执行1次验证且未处理差距就交付
❌ 验证差距报告未保存就交付
❌ 报告末尾无验证执行声明
```

```markdown
|---------|---------|---------|---------|---------|
| 第1次验证 | [日期时间] | [通过/需修订] | [N]项 | [已处理/待处理] |
| 第2次验证 | [日期时间] | [通过/需修订] | [N]项 | [已处理/待处理] |
| 第3次验证 | [日期时间] | [通过/需修订] | [N]项 | [已处理/记录为剩余差距] |

### 验证文件路径

```

## 升级与注意事项

### ⛔ 验证差距处理红线

**以下理由严禁用于跳过差距整改：**
- "时间成本" / "边际价值" / "非关键项"
- "后续版本补充"
- "已披露即可"
- "超过验证迭代次数上限"

**唯一允许的剩余差距类型：**
需要运营主体/第三方提供的事实信息，AI无法自行生成或推断。

**判定测试：**
问自己："这个差距我能否在不需要运营主体提供额外信息的情况下修复？"
- 能 → 整改型，必须修复
- 不能 → 信息缺失型，可披露

对于高风险事项始终包含：

> **⚠️ 法律建议免责声明**：本评估指引基于《个人信息保护法》《数据安全法》《网络安全法》及GB/T 39335-2020等法律法规和标准提供，仅供参考，不构成正式法律意见。对于涉及重大合规风险、监管执法应对、复杂跨境数据传输场景等事项，建议咨询专业个人信息保护律师或法律顾问。

**高风险触发情形**：大规模敏感PI处理、涉及CII的跨境传输、具重大影响的自动化决策、处理未成年人PI（不满14周岁）、监管调查/执法应对、数据泄露事件。
